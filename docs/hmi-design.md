# HMI: design for generation-first operator screens

Status: design, 2026-07-18. Nothing here is implemented; this document is the
plan of record for the feature and the contract its phases will be reviewed
against.

## What we are building, and the one idea that shapes it

IA2 gets a fourth authorable artifact. Today a project holds programs
(`pous/`), devices (`devices/`), wiring (`iomap.toml`) and scheduling
(`tasks.toml`); it will additionally hold operator screens (`hmi/`), rendered
live against the running program and able to write back to it. The feature is
called HMI because that is what the industry calls it, but the deliverable is
narrower and sharper than a general SCADA screen builder: **screens are
generated by agents first and touched up by humans second.**

That ordering is the whole design. Every commercial HMI tool assumes a human
draws screens over days; the agent, if present at all, is a copilot inside a
drawing tool. IA2 inverts this the same way it inverted PLC programming: the
agent owns authorship through a complete API, the human owns intent and
supervision through the IDE, and the artifact lives in the project as a
reviewable file. An agent that has just written the tank-farm program and
wired its iomap already knows every variable, its direction, its device, its
units and its role — it is better placed than anyone to produce the first
correct pass of the screens, seconds after the program compiles.
[Pencil](https://www.pencil.dev/) ("design on canvas, land in code") is the
closest existing shape of this workflow, and we borrow from it deliberately;
the differences are equally deliberate and listed below.

## What the survey settled

Four sources of prior art were examined, and each contributed one settled
decision.

**Pencil (.pen format and MCP tool-face).** Pencil stores a design as a JSON
object tree — typed nodes with ids, flexbox-ish container layout, multiple
fills/strokes/effects — plus three things that make the format *agent-native*
rather than merely machine-readable: document-scoped `variables` referenced
with `$` and resolved through a `themes` axis system; components declared by
marking any node `reusable: true` and instantiated by `ref` nodes carrying
sparse `descendants` overrides; and an editing surface built around bulk
structured operations plus a screenshot/layout-snapshot feedback loop, guided
by loadable guidelines. We adopt all three ideas: our screens are typed node
trees with `$token` references into the project's design tokens, our symbol
library is components-plus-overrides rather than copy-paste, and our agent
loop is "mutate through the API, then look at a rendered screenshot, then
mutate again". We do *not* adopt Pencil's generality — no shaders, mesh
gradients, arbitrary vector paths or free-form text layout. An HMI schema
earns its keep by being small enough that a generated document is almost
always valid and reviewable in a diff.

**ISA-101 / high-performance HMI.** The modern consensus (gray field,
low-saturation base, color reserved exclusively for abnormal or actionable
states, display hierarchy from L1 plant overview down to L4 diagnostic
detail) is not a style preference — it is the operator-safety result the
standard exists to encode. It also happens to be exactly the philosophy the
IA2 design system already implements for the IDE: warm neutrals everywhere,
`--highlight` green only for running/healthy, `--warn` ochre only for
attention, `--destructive` red only for faults, acid green reserved for
agent presence. So the HMI does not get a theming system in v1; it gets the
IA2 token set with an ISA-101 interpretation baked into every built-in
symbol, and generated screens are high-performance-compliant by
construction. This is a genuine differentiator: most tools make ISA-101
compliance the user's homework.

**Ignition Perspective (binding taxonomy).** Perspective's view.json proves
the resource model (a screen is a JSON file at a path; the path is its
identity and its navigation target) and its binding system provides the
taxonomy worth stealing: property-carries-binding, unidirectional by default,
explicitly bidirectional for inputs, with a transform chain (map / format /
expression) on the read path. We implement a deliberately small subset — one
binding source kind in v1 (a program variable), expressions limited to a
pure, non-Turing scalar language, and write-back modeled as *actions* rather
than bidirectional bindings, because writes to a live plant deserve explicit
verbs, confirmation semantics and audit, not silent property sync.

**FUXA and the 2026 generative wave.** FUXA demonstrates that a pure-web
SVG-rendered HMI is fully viable on embedded hardware, and its all-in-one
Node backend demonstrates the architecture we must avoid — IA2 already has a
server, a runtime, a transport and an auth perimeter; the HMI must ride
them, not duplicate them. The generative-HMI market (e.g. Replay's
video-to-React modernization) validates demand but targets brownfield
conversion; nobody ships greenfield generation from live project context,
which is precisely our wedge. One lesson adopted from the AI-HMI literature:
anything an agent generates or recommends on an operator surface must be
explainable and overridable in place — which for us reduces to: screens are
files in the project, diffs are reviewable, and the agent-takeover bar
already tells the operator when a machine is editing.

## The document format: `hmi/<name>.hmi.json`

A screen is one JSON document under `hmi/`, exactly parallel to a graphical
POU under `pous/` — created, read, saved and deleted through the same store
conventions, slash-pathable into folders, identified by its project-relative
slug. The schema lives in the `project` crate as serde types with `#[derive(TS)]`,
so the web renderer, the editor and the generated TypeScript stay in
lockstep with the Rust validator, the same way `FbdProgram` does today.

The shape, abbreviated to its load-bearing parts:

```jsonc
{
  "version": 1,
  "name": "carbonation_overview",
  "title": "Carbonation — Overview",       // operator-facing
  "level": 2,                               // ISA-101 display level 1..4
  "grid": { "w": 1280, "h": 800, "snap": 8 },
  "root": {
    "id": "root", "type": "group", "layout": "absolute",
    "children": [
      { "id": "t1", "type": "symbol", "symbol": "tank",
        "x": 120, "y": 80, "w": 160, "h": 220,
        "props": { "label": "T-201", "unit": "%" },
        "bind": { "value": "level_pct", "alarm": "level_hh" } },
      { "id": "v1", "type": "symbol", "symbol": "valve",
        "x": 320, "y": 260, "w": 48, "h": 48,
        "bind": { "open": "yv201_fb" },
        "action": { "tap": { "kind": "toggle", "variable": "yv201_cmd",
                              "confirm": true } } },
      { "id": "tr1", "type": "trend", "x": 560, "y": 80, "w": 640, "h": 240,
        "series": [ { "variable": "level_pct" }, { "variable": "level_sp" } ],
        "window_s": 300 },
      { "id": "lbl1", "type": "text", "x": 120, "y": 40,
        "text": "Feed section", "style": "section" }
    ]
  }
}
```

Node types are a closed set: `group` (absolute positioning only — flow
layouts were designed, never implemented, and retired by decision: screens
are spatial documents, and absolute coordinates are the truth the arrange
editor, spawn overlays and letterboxing build on; `validate_hmi` warns on
the legacy `vertical`/`horizontal` values),
`text`, `value` (numeric/boolean/string readout with unit and format),
`symbol` (an instance of a library symbol), `trend` (sparkline/strip chart
over the live snapshot history), `alarmbar` (renders the run's
`last_error`/`fault` and per-device health), `button`, `input` (numeric
entry), `nav` (jump to another screen by slug — Perspective's
path-is-navigation), and `shape` (rect/line/polyline for pipes and vessels,
deliberately minimal). Every node has a stable `id`; every visual property
may be a literal or a `$token` reference into the design-token vocabulary
exported from the IDE theme (`$highlight`, `$warn`, `$surface`…), which is
how screens follow light/dark and stay on-palette without ever naming a hex.

**Bindings** are per-node maps from a prop name to a binding spec. The v1
spec is one of: a bare variable name (`"level_pct"`, resolved against the
running program exactly as Monitor resolves names, including
`instance.variable` for multi-PROGRAM runs); or an object
`{ "variable": "...", "expr": "x / 100", "format": "%.1f" }` where `expr` is
a pure arithmetic/comparison expression over the single bound value `x`
(plus named constants), and `format` is printf-ish. Booleans drive symbol
states; numbers drive readouts, fills and trend series. There is exactly one
data source in v1 — the live `VarSnapshot` stream — because that is the one
source that exists; the spec object leaves room for `device_health` and
history queries later without reshaping documents.

**Actions** are the write path and are never implicit. A node may carry
`action.tap` (or `action.commit` for `input`) of kind `write`, `toggle`,
`pulse` (write 1 then 0 after `ms`) or `nav`, each naming a variable and
carrying `confirm: true|false` plus an optional `min`/`max` guard for
numeric entry. Actions call the existing `/api/runtime/write` (or the
edge's `/write`) — the HMI adds no new mutation surface to the runtime, so
everything the safety review already covers (force semantics, failsafe,
agent attribution) remains the single story. The pulse reset is a
**server-side guarantee**: the write request carries `pulse_ms` and the
runtime schedules the 0-write itself, so a closed tab, a suspended
tablet, or a network drop cannot leave a momentary command latched
(issue #22); overlapping pulses on one variable are latest-write-wins,
and `validate_hmi` warns when `ms` exceeds 10 s (that's a toggle).

**Symbols** are the component system, and v1 ships them as a *built-in
library*, not user-defined components: `tank`, `valve`, `pump`, `motor`,
`pipe` (H/V/elbow segments), `gauge`, `indicator`, `setpoint`, plus the
composites `pid_faceplate` and `motor_faceplate`. Each symbol is implemented
once in React/SVG, styled to ISA-101-on-IA2-tokens, and exposes a small
typed prop/bind/state contract (a valve knows `open`, `fault`, `cmd_pending`;
a tank knows `value` 0..100, `alarm`). Project-defined reusable symbols
(Pencil's `reusable`/`ref`/`descendants` pattern) are the designed extension
— the schema reserves `"type": "ref"` — but deferred, because the built-in
set covers the field projects we have and user components are where HMI
builders historically drown in scope.

Validation is a pure function in `project` (`validate_hmi`): unknown node
types, duplicate ids, dangling `nav` targets, bindings that name variables
absent from the project's variable index, actions that write read-only
directions — all surfaced as the same `CheckDiagnostic` shape the compile
path uses, so `cs hmi check` and `/api/project/validate` fold HMI problems
into the existing diagnostics story.

## The expressiveness layer: maps, state color, motion

The v1 vocabulary proved the model but capped what a screen could *say*: nine
symbols, hard-coded colors, no way to express "above 80 °C this number turns
red" without abusing an alarm bind. The expressiveness layer closes that gap
against the commercial web-SCADA baseline (Ignition Perspective's
binding-transform system was the reference) while refusing its scripting —
everything below is still declarative data in the document, validated by
`validate_hmi`, reviewable in a diff.

**Maps.** A binding spec may carry `map`, an ordered list of rules evaluated
after `expr`: `{ "eq": 1, "out": "RUNNING" }` matches exactly,
`{ "min": 80, "out": "alarm" }` / `{ "min": 50, "max": 80, "out": "warn" }`
match a half-open `[min, max)` range, and a condition-less entry is the
catch-all. The first match wins and its `out` string becomes the binding's
output. This is the declarative equivalent of Ignition's map transform, and
it is the *only* way a value becomes a color or a word — no scripts, no
per-client logic to drift.

**Where map outputs land.** The renderer understands a small set of
generic bind keys on top of each node's own: `visible` on every node
(0 hides in Operate; Arrange ghosts it so it stays editable), `color` on
`text`/`value` and on symbols that take an accent (tank liquid, bar fill,
led digits, pipe stream), `text` on `text` nodes (a mapped state label:
0→"STOPPED", 1→"RUNNING"), and `fill`/`stroke` on shapes. Color outputs
speak the token vocabulary — `ok`, `warn`, `alarm`, `info`, `muted`, `fg`,
`agent` map to the design system's CSS variables so screens follow theme
changes; any other string passes through as a literal CSS color as the
deliberate escape hatch. `%s` as a format shows the raw snapshot text,
which is how STRING variables reach a screen.

**Style props.** `text` accepts `color`, `size`, `align`, `weight`;
`shape` accepts `fill`, `stroke`, `stroke_width`, `rx`, `dash` and adds an
`ellipse` kind — enough free-form P&ID drawing to compose what the symbol
library doesn't ship, without opening an arbitrary-SVG door.

**The symbol library, round two.** Seven additions, each earning its place
against a named competitor feature rather than padding the count: `analog`
(the moving analog indicator ISA-101 actually prefers over gauges — scale,
shaded normal band, live pointer, setpoint tick), `bar` (linear fill, h/v),
`led` (headline numeric on a dark plate), `sparkline` (axis-less inline
history), `pipe` (process line whose `flow` bind animates travel, sign =
direction), `fan` (blades spin while running) and `conveyor` (stripes
travel while running). Sixteen total.

**Motion.** Running equipment spins (`pump`, `fan`), flowing lines travel
(`pipe`, `conveyor`), levels ease between samples (tank fill, bar, analog
pointer). Motion is state, not decoration — everything it says is also said
by color/shape, and `prefers-reduced-motion` turns all of it off.

**Binding reliability, hardened.** Tail-name resolution
(`level` matching `feeder.level`) now requires the tail to be *unique* in
the snapshot — two programs both owning a `temp` render "—" instead of
whichever serialized first, and `cs hmi check` warns "exists in N POUs —
qualify as instance.variable" at authoring time. Non-finite results
(`x / 0`, STRING parsed as number) resolve to null, never to "Infinity" on
an operator screen.

## Architecture: who does what

**`project` crate** — schema types (ts-rs exported), store CRUD for
`hmi/*.hmi.json` (mirroring the pou-file functions), `validate_hmi`.

**`server` crate** — REST resources `GET/POST/PUT/DELETE /api/hmi[/{path}]`
following the POU endpoints' shape (project-scoped via `X-IA2-Project`,
`Mutation` SSE events on write so open IDE windows refresh), plus
`/api/hmi/{path}/check`. No rendering server-side; the server stays a JSON
program.

**`apps/web`** — two surfaces sharing one component set. The *runtime view*
(`?hmi=<slug>` route or an HMI tab beside Monitor) renders the document
read-only-plus-actions against `useLastSnapshot()` — the store built
precisely so a tens-of-Hz feed re-renders only its subscribers — and issues
action writes through the existing api helpers with a confirm dialog where
declared. The *editor* is a third graphical editor beside FBD/SFC and reuses
their extracted shell (`useProgramEditor`-family hooks, header, diagnostics
banner): canvas with drag/snap/resize on `grid.snap`, palette of symbols,
inspector for props/bind/action, live "preview with data" toggle. The
renderer is plain React over SVG/DOM — the FBD editor already proves the
canvas patterns, and FUXA proves SVG carries full plants; we need no canvas
library.

**`crates/runtime` + edge** — the operator-panel path. The edge runtime
already streams snapshots (`/events`) and accepts `/write`; it gains the
same `--static-dir` the server has, so a deployed edge box serves the built
web bundle and a shop-floor tablet opens `http://edge:13001/hmi/overview`
directly against local data with no IDE and no dev machine in the loop.
Deploy ships `hmi/` with the project exactly as it ships `pous/` today.
This is deferred to its own phase but designed now so nothing in the format
assumes the IDE server.

**Agent surface** — the reason the feature exists, in three layers. First,
plumbing: `cs hmi list|get|save|delete|check` as thin JSON passthroughs, same
grammar as `cs pou`. Second, generation: `cs hmi generate [--program P]
[--level 2]` calls `POST /api/hmi/generate`, which builds a first-pass
screen *deterministically* from project truth — every mapped variable gets a
node chosen by type and direction (BOOL out → indicator/valve by name
heuristics, REAL in → readout or setpoint pair, iomap devices group into
sections, alarm-ish names(`*_hh`, `*_alm`, `fault*`) wire the alarmbar), laid
out on the grid in sections. Deterministic generation gives any agent — and
any human clicking a button — a correct, boring baseline in one call; the
*creative* pass (an LLM agent restructuring toward the process topology,
adding trends where dynamics matter, writing a faceplate flow) then edits
the document through the same CRUD it already knows, exactly like it edits
FBD JSON today. Third, feedback: the skill documents the loop the Pencil
tool-face teaches — save, render `?hmi=slug` headless (agent-browser is
already the house pattern), screenshot, self-review against the ISA-101
checklist in the skill, iterate. A `symbols` catalog endpoint
(`GET /api/hmi/symbols`, name + prop/bind contract per symbol) keeps the
agent's knowledge of the palette in sync with the code instead of in prose.

## Safety semantics

Reads are unrestricted — the snapshot is already public to every IDE
surface. Writes get three fences, all in v1: actions are the only write
path and each declares itself in the reviewable document (a diff shows
exactly which screens can write what); `confirm: true` renders an explicit
confirmation with the target value (default for anything not explicitly
marked safe); numeric guards clamp at the client *and* are re-validated by
`validate_hmi` against declared bounds. Agent-authored edits to HMI files
travel through the same agent-session attribution as everything else, so
the acid bar announces machine authorship in the IDE while it happens.
Deeper role separation (operator vs engineer login on the edge panel) is
explicitly out of scope for v1 and noted as the first hardening item once
panels face real operators, consistent with the edge's current
ssh-perimeter trust model.

Two doctrines from field use (RFC #33), held as design law: **Stop never
requires confirmation** — a stop action's `confirm` should be false, and
adding a dialog in front of "make it stop" is a defect, not caution. And
**a screen stop button is not an e-stop and must not cosplay as one** —
label it Stop/Abort, style it as a control, and document the physical
e-stop as the real safety device; a soft-PLC HMI never claims a safety
function it cannot deliver.

## Phasing

**P0 — format + runtime view + agent plumbing.** Schema/store/validate in
`project`; REST + symbols catalog + deterministic `/generate`; `cs hmi`
family; runtime view rendering the built-in symbol set with bindings,
actions, trend and alarmbar; skill section with the generate→screenshot→
iterate loop. Exit test: from a fresh clone, one agent command chain takes
`carbonation_demo` from no-HMI to a rendered live overview screen driven by
the running simulation, and the api.md coverage gate plus a schema round-trip
test hold.

**P1 — the human editor.** Canvas editor as the fourth graphical editor on
the shared shell: palette, inspector, drag/snap, live-preview. Exit test:
a human can produce and adjust the same screen with no JSON editing, and
agent and human edits interleave losslessly through the same file.

**P2 — the edge operator panel.** `--static-dir` on the runtime, deploy
ships `hmi/`, panel route binds to the edge's own `/events` + `/write`;
kiosk notes in edge-deploy.md. Exit test: SV660N bench box serves its own
screen to a tablet with the dev machine offline.

**P3 — depth, driven by field use.** Project-defined `ref` symbols with
overrides; history-backed trends once the edge DAQ story exists; multi-source
bindings (`device_health`, edge system stats); alarm acknowledgment flow;
and **generation-as-proposal** (accepted from RFC #33-B): `generate`
against an existing screen emits a reviewable op-diff the human accepts in
the IDE — riding the mutation/spawn machinery — instead of `--force`
overwriting curation. Provenance markers were considered and rejected
(ownership of a generated node rots the moment a human drags it; a
reviewed diff doesn't).

## Open questions (tracked, not blocking P0)

Whether trend history should buffer client-side only (v1: yes, window-limited)
or the runtime should retain a ring buffer per variable; whether `expr`
grows cross-variable references (v1: single-variable on purpose — the moment
expressions span variables, screens start hiding logic that belongs in a
POU); and where the line sits between symbol props and full user components
when P3 opens that door. Each is answerable with field evidence from P0/P2
deployments rather than speculation now.
